1. Introduction

Purpose & Scope

SRS 用于定义 SyncBridge 的目标、功能、性能与约束，指导开发、测试与后续维护。平台聚焦客户端与开发者的需求澄清全流程。

Conventions & References

Times New Roman 11pt、分级标题；优先级 High/Medium/Low。参考 IEEE 830-1998、课程模板、团队讨论。

结构

第 2 章总体描述，第 3 章功能与非功能需求，附录含术语、状态定义与 TBD 清单。

2. Overall Description

架构

Web 客户端 + 后端服务 + 数据库；外部邮件服务；未来可集成外部 AI（当前不在范围）。

系统环境与角色

角色：Client（提交/澄清需求）、Developer（分析/执行需求）；通过浏览器访问，后台调用邮件服务。

用户特征

Client：非技术、关注业务价值；Developer：技术背景，评估可行性并提出澄清问题。

假设/依赖

稳定数据库、SMTP、文件存储、第三方 API；稳定基础设施与运维支持；用户具备基础系统素养。

2.3 Use Cases

UC-1 提交需求 (Client)

填写表单 → 校验 → 创建需求（初始状态 Preview）→ 分配 ID → 确认；异常时提示错误。

UC-2 更新状态 (Developer)

查看详情 → 选择新状态 → 校验工作流 → 更新并记录历史；无权限或非法流转则拒绝。

UC-3 发送消息 (双方)

打开需求 → 输入消息/附件 → 校验与存储 → 展示于历史；附件超限则提示。

UC-4 注册/登录

注册创建账号（唯一标识）或登录验证凭证；失败时返回错误。

UC-5 通知发送 (系统)

事件触发 → 找到收件人 → 按规则发送通知；失败记录以便排障。

3.1 Functional Requirements

认证 (FR-1~4)

  • 注册/登录，支持退出与找回密码。

需求提交/管理 (FR-5~8)

  • 提交、查看、编辑（受状态限制）。
  • 详情包含描述、状态与讨论。

沟通与附件 (FR-9~13)

  • 消息发布与历史展示。
  • 附件 10MB 限制，超限提示使用外链。

状态机 (FR-14~19)

  • 状态：Preview/Available/In Progress/Rewrite/End/Error。
  • 合法流转：Preview→Available→In Progress→Rewrite↔In Progress→End/Error；Error/End 终止。
  • 互相确认后方可 End；记录状态历史。

通知 (FR-20~23)

关键事件发送通知；紧急 5 分钟提醒，普通 48 小时提醒；可配置基础偏好。

审计 (FR-24~26)

记录提交、编辑、状态与消息操作；授权用户可查看，支持导出。

License (FR-27~31)

输入并验证 License；无有效 License 限制核心功能；反馈失效原因；记录 License 事件。

3.2 Non-Functional

性能与可靠性

响应 < 3 秒，支持 100 并发；年可用性 99.5%；异常不损坏数据并记录错误。

可用性与安全

关键操作不超过三步；中英双语；基于角色的访问控制；密码不明文存储。

可维护/可扩展/可移植

易扩展，无需大改即可新增功能；有充分文档；可随用户与需求量增长扩展；常见浏览器和标准服务环境可用。

可审计性

全链路可追踪；审计记录包含动作、时间戳、操作者。

Glossary & Status

关键术语

Requirement、Requirement ID、Message、File Upload、Email Notification、Audit Log、Traceability、Session、Authorization、High Concurrency、Accessibility、External Dependency。

状态定义

  • Preview: 草稿，可编辑，未对开发者公开。
  • Available: 发布，可被认领。
  • In Progress: 已被单一开发者认领，处理中。
  • Rewrite: 协商/修订阶段，可返回 In Progress。
  • End: 双方确认完成，终止。
  • Error: 无法解决的情况，终止。
